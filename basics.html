Panels, boxes, widgets highlighted sections, basically anything that could contain text and a background of some sort. Some sites donâ€™t have them some do and web applications nearly alway have them. They also appear across a number of contexts so defining a pattern is of great value.

Standard box

Each 'box' consists of both margin and padding. In other css markup systems we see the use of the classname 'pod' like in SMACSS. Here our 'box' is abstracted away into 'content'.
It's also important to realise then some cases you will need to use a 'div' instead of a 'section'

HTML

<section class="content">
    … general markup …
</section>

LESS

.box() {
    margin: 0;
    padding: @interline @block @em;
    .box-sizing( border-box );
    }
.box( size;  @unit: 1; ) {
    width: ( @grid-span * @unit );
    }
.content {
    .box();
    }

Box inset

If you need to override the padding of the outer box you can use the 'box‑inset' class. This is useful because you don't need to know what the padding is as it is set by a variable in '.box'.

Tip: Don't use more than a single decedent selector. Keep it short!

HTML

<section class="content">
    <header>
        … header text …
    </header>
    <p>
        … paragraph text …
    </p>
</section>

LESS

.box-inset() {
    margin: -(@interline) -(@block) -(@em);
    padding: @interline/2 @block;
    .box-sizing( border-box );
    }
.content header {
    .box-inset();
    }

Widget Panels

A widget is defined through the 'panel' (parametric) mixin and contains a 'box' mixin. The 'panel' is in effect an extention of 'box' with some specific properties that a widget could use. The 'widget' then gets its own class definition with panel and optionaly some parent classes to give context.

Pattern matching can be very useful for creating a number of styles variations. The downside the css output can get rather large. So keep an eye on your CSS when nesting mixins like this.

HTML

<section class="dashboard">
    <div class="widget">
        <header>
            … header text …
        </header>
        <form>
            … form …
        </form>
        <footer>
            … footer markup …
        </footer>
    </div>
</section>

<aside>
    <div class="widget">
        … widget markup …
    </div>
</aside>

LESS

.panel() {
    .box();
    .border-radius( @em / 2 );
    header, footer {
        .box-inset();
        }
    }
.panel( dark; ) {
    .gradient( @grey6, @grey7 );
    }
.panel( light; ) {
    .gradient( @grey0, @grey1 );
    }
.panel( bordered; ) {
    border: @l solid @grey1;
    }
.panel( @color1; @color2; ) {
    .gradient( @color1, @color2 );
    }

.dashboard{
    .panel( light );
    }
.widget{
    font-family: @typeface-brand;
    .content &   { .panel( bordered ); }
    .dashboard & { .panel( dark ); }
    aside &      { .panel( @grey2; @grey3 ); }
    }

The weakest link when it comes to the web is layout. Help is on the way but it's not here yet. For those building enterprise based web applications it will be a much longer wait. Sometimes a predefined grid can work but usually it's more trouble then it's worth. So defining a pattern that can map to a grid can be way more powerful.

We can think of grids in a number of ways. Either in a sequence of columns or in proportional sections like halves, thirds, quarters and so on. You can map one to the other so think about this when implementing a layout in terms of a grid.

Implementing a web based grid today is largely a bungled affair in that you only need to reflect the parts that visibly align with the underlying grid. Almost all of the grid is never seen by the user, so you don't need to implement that part. Were grid systems gain traction is that they give designers and developers a common vocabulary. This is useful, inserting a full grid system usually isn't. Until the flex module is available across the actively used browsers we need an alternative light weight way to represent the grid that is obvious and easy to implement.

Grid sizing and offset

On the web a grid is effective a width unit. Depending on the design you can determin what that unit is and how it is subdevided.

HTML

<div class="widget">
    … general markup …
</div>

LESS

.grid-span( @multiplier; @offset: 0;  ){
    width: ( ( @multiplier * @grid-span ) - ( @offset * @grid-span ) );
}
.grid-offset( @multiplier ) {
    margin-left: ( @multiplier * @grid-span );
    }
.widget{
    @offset: 1; /* number of grid units */
    .grid-offset( @offset );
    .grid-span( 3, @offset );
}

Grid columns

We don't have flex box, but this will do. In fact this very page uses this solution.

HTML

<div class="widget">
    … general markup …
</div>
<div class="widget">
    … general markup …
</div>
<div class="widget">
    … general markup …
</div>

LESS

.box-to-grid( @columns; @gutterSize: @block ){
    @responsiveColumn: ( 100 / @columns );
    @columnAdjust: ( ( ( @columns - 1 ) * @gutterSize ) / @columns ) ;
    float: left;
    display: inline-block;
    margin-left: @gutterSize;
    width: calc( ~"@{responsiveColumn}% - @{columnAdjust}" ) ;
    &:nth-of-type( @{columns}n+1 ){
        clear: left;
        margin-left: 0;
    }
}
.widget{
    .box-to-grid(3);
}
